# 微信小程序本地存储适配设计

- 日期：2026-05-04
- 状态：设计确认稿
- 范围：微信小程序本地存储 MVP 适配

## 1. 背景

当前 `shounian-miniapp` 已经具备微信小程序主体结构，核心体验集中在今日页：

- 写一功 / 写一过
- 将纸符收入功匣 / 过匣
- 查看匣中符文
- 封存今日并进入归档

现有记录逻辑已经使用 `wx.getStorageSync` 与 `wx.setStorageSync` 做本地存储，但仍缺少正式小程序运行所需的工程配置、数据结构版本约束、跨日期轮转和异常数据兜底。

本设计用于把项目先适配成一个不依赖登录、不依赖授权、不依赖云端的小程序 MVP。

## 2. 目标

第一阶段目标是让小程序在微信开发者工具和真实微信客户端中，以纯本地数据方式稳定运行。

用户应能完成以下路径：

1. 第一次打开小程序，看到今日空记录。
2. 当天记录功符 / 过符，并可封存今日。
3. 第二天再次打开时，自动进入新的一天。
4. 前一天有记录时自动进入归档，后续可在归档页看到。
5. 本地数据异常时页面不崩溃，尽量恢复为可用状态。

## 3. 非目标

本阶段不做以下内容：

- 微信登录
- 获取用户头像、昵称或手机号
- 云开发、云数据库、云函数
- 多设备同步
- 账号体系
- 远程备份
- 社交排行榜或公开分享具体记录

这些能力应在本地 MVP 稳定后再评估。

## 4. 微信能力边界

本地存储能力不需要用户授权。

本阶段只依赖以下微信小程序基础 API：

- `wx.getStorageSync`
- `wx.setStorageSync`
- `wx.showToast`
- 页面生命周期 `onShow`
- 页面跳转 `wx.navigateTo` / `wx.navigateBack`

因此，第一版用户打开即可使用，不出现登录弹窗，也不索取个人信息。

## 5. 数据模型

本地存储使用单一主 key：

```js
const STORAGE_KEY = "shounian-box.records.v1";
```

建议记录结构：

```js
{
  version: 1,
  today: {
    dateKey: "2026-05-04",
    gong: [
      {
        text: "今天做对了什么",
        createdAt: 1777852800000
      }
    ],
    guo: [],
    sealed: false,
    sealedAt: null
  },
  archive: [
    {
      dateKey: "2026-05-03",
      gong: [],
      guo: [],
      sealed: true,
      sealedAt: 1777766400000
    }
  ]
}
```

字段约定：

- `version`：本地数据结构版本，当前为 `1`。
- `today`：当前自然日记录。
- `archive`：已归档日期列表，新日期排在前面。
- `dateKey`：自然日标识，格式为 `YYYY-MM-DD`。
- `gong` / `guo`：纸符数组。
- `text`：用户填写的文字，可为空字符串。
- `createdAt`：记录创建时间戳。
- `sealed`：该日是否已封存。
- `sealedAt`：封存时间戳；未封存时为 `null`。

## 6. 日期轮转规则

读取本地数据时统一经过 `normalizeRecords` 处理。

处理规则：

1. 没有本地数据时，创建今天的空记录。
2. 本地数据缺少 `version` 时，按当前版本补齐。
3. `today.dateKey` 等于今天时，直接使用当前 today。
4. `today.dateKey` 早于今天时：
   - 如果 today 有功符或过符，将其归档。
   - 如果 today 没有任何记录，不写入归档。
   - 创建新的 today。
5. `today.dateKey` 晚于今天时，视为设备时间变更或异常数据：
   - 保留 archive。
   - 创建今天的空记录。
6. archive 中同一 `dateKey` 只保留一条，以最新写入为准。

本阶段按设备本地时间生成日期，不额外处理跨时区同步。

## 7. 模块边界

### 7.1 `utils/storage.js`

职责保持简单：

- 安全读取本地存储。
- 安全写入本地存储。
- 捕获微信 storage 异常。

不在该模块写业务逻辑。

### 7.2 `utils/date.js`

职责：

- 生成 `dateKey`。
- 格式化展示日期。
- 提供日期比较所需的轻量函数。

### 7.3 `services/record-service.js`

职责：

- 创建空日记录。
- 读取并规范化 records。
- 保存 records。
- 添加功符 / 过符。
- 封存 / 解封今日。
- 归档去重。
- 计算今日评语。

跨天逻辑应放在此模块，页面层不直接判断日期轮转。

### 7.4 页面层

页面继续通过 `loadRecords()` 获取可用数据，不关心本地数据是否刚刚迁移、修复或跨天轮转。

## 8. 页面影响

### 8.1 今日页

`pages/today/index` 保持主入口职责：

- `onShow` 调用 `refresh()`。
- `refresh()` 读取规范化后的 records。
- 若跨天轮转发生，页面自然展示新 today。

### 8.2 今日卡片页

`pages/summary/index` 仍展示当前 today。

如果 today 未封存，也允许展示当前状态；分享卡片生成暂不在本阶段实现。

### 8.3 归档页

`pages/archive/index` 展示 archive 列表。

本阶段可继续保持摘要列表；后续再增加点击某天查看详情。

## 10. 异常处理

本地数据异常时采用温和降级：

- storage 读取失败：返回默认空数据。
- records 不是对象：返回默认空数据。
- `today.gong` / `today.guo` 不是数组：修复为空数组。
- archive 不是数组：修复为空数组。
- 单条 slip 缺少 `createdAt`：补当前时间。
- 单条 slip 缺少 `text`：补空字符串。

修复后的数据在下一次保存时写回本地。

## 11. 验证方式

本轮验证以定向验证为主：

1. 静态检查关键文件结构是否符合小程序约定。
2. 手动构造不同 storage 数据，验证 `loadRecords()` 输出。
3. 在微信开发者工具中导入项目并预览今日页。
4. 手动验证：
   - 首次打开
   - 写一功
   - 写一过
   - 封存
   - 解封
   - 归档展示
   - 模拟跨天

如果当前环境无法直接运行微信开发者工具，应明确说明未完成真机 / 模拟器视觉验证。

## 12. 后续阶段

本地 MVP 稳定后，再考虑：

1. 归档详情页。
2. 分享卡片生成。
3. 数据导出或备份。
4. 微信登录与云端同步。
5. 多设备一致性。

后续云端路线应以不破坏本地数据模型为前提。
